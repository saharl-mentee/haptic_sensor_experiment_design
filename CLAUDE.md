This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

Design-of-experiments (DOE) tooling for a haptic force sensor (LC circuit whose resonant frequency shifts as a metal sheet deforms). The code helps choose **which of the 135 possible factor combinations to physically build** (~20–25 sensor builds affordable) and will later analyze the measurements with ANOVA. It is not an app or library — it is a decision/analysis toolkit run as scripts.

The user (Sahar) is a hardware engineer with little statistics background: **all code here is deliberately educational** — long plain-language docstrings explaining the statistics, printed explanations alongside numbers, small readable implementations instead of black-box libraries (pyDOE3/dexpy can't handle multi-level categorical D-optimal designs anyway). Preserve that style in any change.

## Commands

```bash
source ~/.venvs/haptic_doe/bin/activate   # venv lives OUTSIDE the repo (WSL home, /mnt/c is slow)
cd doe
python compare_designs.py                 # the Phase 0 driver, ~40 s, writes ../results/
python optimal.py                         # any module runs standalone as a demo/self-test
```

There is no test suite, linter, or build. Verification = run `compare_designs.py` end-to-end plus the inline asserts (orthogonality checks in `taguchi.py`, the Fedorov self-test in `optimal.py` which must recover the 3×3 full factorial at N=9).

Scripts must be run **from inside `doe/`** (modules import each other by bare name; `results/` is resolved relative to the file).

## Architecture

Dependency chain: `factors.py` → `model_matrix.py` → {`taguchi.py`, `optimal.py`} → `metrics.py`, `power_sim.py` → `compare_designs.py` (driver).

- `factors.py` is the single source of truth: factor levels (placeholders C1–C5, FoamA–C, T1–T3, MetalA–C until real materials are known) and the power-simulation assumptions (`NOISE_CV`, `EFFECT_SIZE`). Everything downstream is generic over it.
- All modules share the `FACTORS` dict **by object reference** — to swap factors temporarily (as `optimal._self_test` does), mutate it in place (`clear()` + `update()`), never reassign.
- A "design" is just a pandas DataFrame with one row per sensor build and one column per factor. `metrics.scorecard()` and `power_sim.power_table()` accept any such DataFrame, including hand-edited CSVs.
- `power_sim` rotates the injected effect across every level of a factor and averages — this is deliberate (fairness to unbalanced designs); don't "simplify" it back to a fixed level.
- `results/` is fully regenerated by the driver and is committed on purpose (the `design_*.csv` files are the lab run lists).

## Project state and roadmap

Phase 0 (this code) is done. Agreed next phases, in order, once the user picks a design/N:
1. **Phase 1**: final design generator → randomized build/test run sheet + raw-data CSV template (row = run, cycle, mass, resonant frequency).
2. **Phase 2**: analysis pipeline — per-build response extraction (sensitivity = slope of f vs force, dynamic range, repeatability), Type-II ANOVA, effect plots, confirmation build.
3. **Phase 3**: position-sensing study (all 5 coils × winning foam/metal stack, pin-force at grid positions, repeated-measures) + paired mini-test of the 2 mirror-winding coil variants.

Domain decisions already settled (don't re-open without the user): each factor combination is a complete new sensor build (no split-plot); the mass sweep ×3 cycles is within-run measurement, not DOE runs; the metal factor bundles material+thickness (confounded by design); mirror-winding coils are excluded from the main DOE.
